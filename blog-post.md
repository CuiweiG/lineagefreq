# Tracking Variant Replacement in Three Lines of R

In late 2023, the JN.1 sublineage of SARS-CoV-2 rose from undetectable
levels to over 80% of sequenced isolates in the United States within
roughly ten weeks. Public health agencies needed to know how fast it was
growing, when it would dominate, and whether their sequencing volumes
were sufficient to track it. These questions recur with every new
variant — and they are, at their core, questions about lineage frequency
dynamics that can be answered with well-established statistical models.

`lineagefreq`, now on CRAN, is an R package for estimating growth
advantages, fitting frequency trajectories, and generating short-term
forecasts from genomic surveillance count data. It was designed around a
specific principle: the path from raw sequencing counts to actionable
estimates should be short, reproducible, and honest about uncertainty.

## The Statistical Foundation

When two pathogen lineages co-circulate, their relative frequencies over
time follow a logistic trajectory under the assumption of constant
fitness differences. The multinomial logistic regression (MLR) model
formalises this: observed sequence counts at each time point are
multinomially distributed, with log-linear frequency trajectories
parameterised by lineage-specific growth rates. The growth rate
difference between a focal lineage and a reference lineage can be
converted to a relative effective reproduction number Rt using the
generation time (Piantham, Linton, and Nishiura 2022).

This framework is not new. What `lineagefreq` contributes is a unified,
well-tested implementation with built-in evaluation infrastructure that
lets analysts move from data to validated forecasts without assembling
bespoke pipelines.

## From Counts to Growth Advantages

The core workflow begins with
[`lfq_data()`](https://CuiweiG.github.io/lineagefreq/reference/lfq_data.md),
which validates and standardises raw count data, computes per-timepoint
frequencies, and flags unreliable periods with low sequencing volume:

``` r
library(lineagefreq)
data(cdc_sarscov2_jn1)

x <- lfq_data(cdc_sarscov2_jn1, lineage = lineage, date = date, count = count)
```

Model fitting uses a single interface regardless of the underlying
engine:

``` r
fit <- fit_model(x, engine = "mlr")
```

Growth advantages are extracted in whichever parameterisation the
analyst prefers — raw growth rates, relative Rt, selection coefficients,
or doubling times:

``` r
growth_advantage(fit, type = "relative_Rt", generation_time = 5)
```

On CDC surveillance data covering the BA.1 to BA.2 transition, the model
recovers a relative Rt of approximately 1.34, consistent with published
household transmission estimates from Denmark (Lyngse et al. 2022).

## Five Engines, One Interface

Not every surveillance question calls for the same model. `lineagefreq`
provides five estimation engines behind a common
[`fit_model()`](https://CuiweiG.github.io/lineagefreq/reference/fit_model.md)
interface:

| Engine     | Type            | Use case                            |
|------------|-----------------|-------------------------------------|
| `mlr`      | Frequentist     | Quick estimates, single location    |
| `hier_mlr` | Hierarchical    | Multiple locations with sparse data |
| `piantham` | Frequentist     | Biologically interpretable Rt       |
| `fga`      | Bayesian (Stan) | Full posterior uncertainty          |
| `garw`     | Bayesian (Stan) | Time-varying fitness landscapes     |

The hierarchical MLR engine applies DerSimonian–Laird random-effects
meta-analysis across locations, shrinking noisy site-specific estimates
toward the pooled mean. This is particularly valuable for early-phase
surveillance, when individual countries or states may have few
sequences.

## Honest Forecasting

Frequency forecasts are generated by propagating uncertainty from two
sources: parameter estimation error (via the Fisher information matrix)
and multinomial sampling variability at the effective weekly sequencing
volume:

``` r
fc <- forecast(fit, horizon = 28, n_sim = 1000)
autoplot(fc)
```

The result is a set of prediction intervals, not point forecasts. This
matters operationally — a 4-week forecast with wide intervals
communicates something fundamentally different from one with narrow
intervals, and downstream decisions (surge planning, vaccine updates)
should reflect that distinction.

## Built-in Backtesting

Perhaps the most important feature for methodological credibility is
[`backtest()`](https://CuiweiG.github.io/lineagefreq/reference/backtest.md),
which implements rolling-origin out-of-sample evaluation following the
framework of Abousamra, Figgins, and Bedford (2024):

``` r
bt <- backtest(x, engines = c("mlr", "piantham"), horizons = c(7, 14, 21, 28))
sc <- score_forecasts(bt, metrics = c("mae", "rmse", "coverage", "wis"))
compare_models(sc)
```

This procedure repeatedly fits the model on historical data, generates
forecasts, and compares predictions to held-out observations. It
prevents the common mistake of reporting in-sample fit as forecast
accuracy. On the BA.2 transition data, 2-week MAE runs approximately 4
percentage points, rising to 8 percentage points at 4 weeks — credible
performance for a model with no epidemiological covariates beyond
sequence counts.

## Surveillance Utilities

Two additional functions address practical surveillance operations.
[`sequencing_power()`](https://CuiweiG.github.io/lineagefreq/reference/sequencing_power.md)
computes the sample size required to detect a variant at a specified
prevalence with a given confidence level — answering the recurrent
question of whether a country’s sequencing volume is adequate.
[`summarize_emerging()`](https://CuiweiG.github.io/lineagefreq/reference/summarize_emerging.md)
scans all lineages for statistically significant growth trends, flagging
those exceeding a frequency threshold.

## Included Data

The package ships with four datasets: two simulated (SARS-CoV-2 US 2022
dynamics and influenza H3N2 clades) and two real (CDC biweekly data
covering the JN.1 emergence and the BA.1-to-BA.2 transition). The real
datasets enable immediate, reproducible validation against published
epidemiological estimates.

## Relationship to Existing Tools

Several groups have implemented multinomial logistic models for variant
frequency estimation. The Nextstrain/Hubbard group’s `evofr` package
provides Bayesian renewal models in Stan. CDC’s internal pipelines use
related approaches. `lineagefreq` is not a replacement for these
specialised tools — it is a lightweight, CRAN-distributed alternative
aimed at surveillance teams, academic labs, and public health analysts
who need a self-contained solution with minimal setup overhead.

The package follows tidyverse conventions throughout. Broom methods
(`tidy()`, `glance()`, `augment()`) ensure that model outputs integrate
directly into existing analytical pipelines. The engine registration
system allows third-party models to be plugged into the same
[`fit_model()`](https://CuiweiG.github.io/lineagefreq/reference/fit_model.md)
interface, following the design philosophy of `parsnip`.

## Installation and Resources

``` r
install.packages("lineagefreq")
```

Four vignettes cover the core workflow, real-data case studies, model
comparison, and end-to-end surveillance pipelines. Source code is at
[GitHub](https://github.com/CuiweiG/lineagefreq).

The underlying statistical models are mature and well-characterised. The
contribution here is engineering: a clean, tested, documented
implementation that lets an epidemiologist go from raw sequence counts
to growth estimates, forecasts, and power calculations in a single R
session.
